# 申請ツール 業務要件

## 1. プロジェクト目的

ITサービス依頼（機器貸与・返却）の申請業務をSalesforceから専用フォームへ切り出し、申請の1/5を占める不要・不備申請を申請段階で排除することで、情報システム部への問合せ工数を削減する。

## 2. 対象業務

| 申請種別 | 対象機器 |
|----------|----------|
| 機器貸与 | PC／通信機器（スマホ・モバイルWi-Fi）／周辺機器 |
| 機器返却 | PC／通信機器／周辺機器（延長申請は返却フローに統合） |

## 3. 認証・アクセス制御

| 項目 | 内容 |
|------|------|
| 認証方式 | Microsoft Entra ID（OIDC/OAuth2） |
| 対象ユーザー | 全社員（ゲストはユーザー権限払い出しで対応） |
| アクセス制限 | 社内VPN経由のみ |
| EntraID連携（AWS） | OIDCリダイレクト方式（アプリ→Entraへリダイレクト、JWTトークン検証） |
| EntraID連携（Microsoft） | （詳細TBD） |

## 4. 承認フロー方針

承認フローは設けない。申請フロー内のバリデーションで不備・不要申請を排除し、通過した申請は情報システム部へ自動通知される。予算承認は本ツールのスコープ外（別途申請フローで対応）。

## 5. 権限・役割

| ロール | 対象者 | 役割 |
|--------|--------|------|
| 申請者 | 全社員 | 機器貸与・返却の申請 |
| 管理者 | 情報システム部 | 申請受付・対応・ダッシュボード管理 |
| システム管理者 | （詳細TBD） | マスタ設定・ユーザー管理・ログ閲覧 |

- 申請者向けのステータス確認・履歴閲覧はフェーズ1ではSalesforce側で対応（スコープ外）
- フェーズ2でEntraIDのユーザーIDをキーに申請者向けステータス確認画面を追加検討

## 6. 外部システム連携方針

現行はSalesforce上のワークフローで機器申請を含む業務全体を管理している。本プロジェクトでは機器申請部分を新フォームへ切り出すが、SF上に残る別申請・ワークフローとの整合性を保つためデータ連携が必要。

| # | 連携先 | 方向 | 用途 |
|---|--------|------|------|
| 1 | Salesforce | 書き込み | 申請データをSF側ワークフローへ連携 |
| 2 | マスタデータ（TBD） | 読み取り | 社員情報・納品先マスタの参照 |
| 3 | ITAM | 双方向 | 資産情報参照・資産管理入力の自動化 |

## 7. 開発・運用方針

| 項目 | 内容 |
|------|------|
| インフラ | AWS（有力）または Azure。複雑な申請分岐ロジックを制約なく実装するためスクラッチ開発を選択。長期的な安定運用・保守性を重視。 |
| 開発体制 | 社内開発（情報システム部・担当者1名） |
| 対応時間 | 業務時間内（9:00〜16:00）。16:00以降の申請・問合せ・障害対応は翌営業日に対応 |
| 問合せ窓口 | メールまたはTeamsチャネル（詳細TBD） |
