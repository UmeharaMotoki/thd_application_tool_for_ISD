# 申請ツール 業務要件

## 1. 認証・アクセス制御

| 項目 | 内容 |
|------|------|
| 認証方式 | Microsoft Entra ID（OIDC/OAuth2） |
| 対象ユーザー | 全社員（ゲストは対象外） |
| アクセス制限 | 社内VPN経由のみ |
| EntraID連携 | OIDCリダイレクト方式（アプリ→Entraへリダイレクト、JWTトークン検証） |
| ゲスト対応 | 将来的にEntra B2Bで対応可能（現時点はスコープ外） |

## 2. 申請フロー

### 目的

申請の1/5を占める不要・不備申請を申請段階で排除し、ITチームへの問合せ工数を削減する。

### 事前ヒアリング（申請フォーム誘導）

申請開始時に簡易ヒアリングを行い、内容に応じて適切なフォームへ誘導する。

| ヒアリング項目 | 分岐内容 |
|----------------|----------|
| 申請種別 | 貸与 / 返却 |
| 機器種別 | PC / 通信機器 / 周辺機器 |
| 申請台数 | 1台（通常フォーム）/ 複数台（大量申請フォームへ分岐） |
| 申請理由の概要 | 新規入社 / プロジェクト / 故障交換 / その他 |

### 申請種別

| 大分類 | 小分類 |
|--------|--------|
| 機器貸与 | PC／通信機器（スマホ・モバイルWi-Fi）／周辺機器 |
| 機器返却 | PC／通信機器／周辺機器（延長申請は返却フローに統合） |

### 入力項目（共通）

| 項目 | 備考 |
|------|------|
| 申請者情報 | 氏名・社員番号 |
| 利用者情報 | 氏名・社員番号（申請者と異なる場合） |
| 納品先 | 社内拠点のみ（自宅不可） |
| 希望納品日 | （基準日以内のみ受付） |

### 下書き保存

| 項目 | 内容 |
|------|------|
| 機能 | 申請途中の内容を一時保存できる |
| 保存期間 | 1週間（期限切れで自動削除） |
| 対象 | 申請者本人のみ参照可能 |

### 申請不可条件（バリデーション）

以下の条件を満たさない場合、申請を送信不可とする。

| 条件 | 内容 |
|------|------|
| 必須情報不足 | 社員番号など資産管理上必要な項目が未入力 |
| 対応不可の納品先 | 納品先が自宅など対応外の場所 |
| 期限超過 | 希望納品日が申請日から1ヶ月以上先は受付対象外（半年先などの申請が散見されるため） |
| 対応不可内容 | ITチームが対応できない機器・内容が含まれる場合 |
| 申請台数 | ハードブロックは設けず、事前ヒアリングで台数・理由を確認してからフォームへ誘導 |
| 大量申請 | 一定台数以上（基準はTBD）の場合は大量申請フォームへ分岐し、理由の詳細入力または情シスへの事前確認を必須とする |
| 一定期間内の申請数制限 | 同一申請者による短期間の連続申請を監査ログで検知（ハードブロックではなくアラートで対応） |

### 機器別 貸与判定分岐

#### PC

| 分類軸 | 分岐内容 |
|--------|----------|
| 社員属性（職種） | 管理社員／技術社員／その他 |
| 社員属性（雇用形態） | 役員／一般社員／派遣社員／外部委託 |
| M365ライセンス | 上記属性の組み合わせで決定（詳細TBD） |
| ネットワーク利用 | 社内のみ／社内＋外部／外部のみ |
| 備考 | 技術社員は顧客ネットワーク環境での利用ケースあり |

#### 通信機器（スマホ・モバイルWi-Fi）

| 分類軸 | 分岐内容 |
|--------|----------|
| カメラ | あり／なし |
| アカウント種別 | 個人特定アカウント／個人特定しないアカウント |

#### 周辺機器（マウス・ヘッドセット・ケーブル・モニター・キーボード等）

- 貸与・返却の記録のみ。詳細な分岐なし。

## 3. 承認フロー

承認フローは設けない。申請フロー内のバリデーションで不備・不要申請を排除し、通過した申請は自動的に情報システム部へ連絡される。

| 項目 | 内容 |
|------|------|
| 承認者 | なし |
| 申請後の流れ | 情報システム部へ通知（詳細は「通知」セクション参照） |
| 予算承認 | 本ツールのスコープ外（別途申請フローで対応） |

## 4. 通知

通知方式はメールのみ（現行システムと同じ挙動を維持）。  
SalesforceのDBとAPIで連携するため、Salesforce側のメール機能はトリガーされない。新システム側でメール送信を実装する。

| タイミング | 送信先 | 内容 |
|------------|--------|------|
| 申請完了時 | 申請者・情報システム部 | 受付完了通知 |
| 対応完了時 | 申請者 | 対応完了通知 |
| その他 | （詳細TBD） | |

| 項目 | 内容 |
|------|------|
| 送信方式 | Microsoft Graph API（M365）またはSMTP |
| 送信元 | （詳細TBD） |

## 5. 権限管理

| ロール | 対象者 | できること |
|--------|--------|-----------|
| 申請者 | 全社員 | 申請フォームの入力・送信のみ |
| 管理者 | 情報システム部 | ダッシュボード閲覧・申請データ管理・ステータス更新 |
| システム管理者 | （詳細TBD） | マスタ設定・ユーザー管理・ログ閲覧 |

- 申請者向けの申請状況確認・履歴閲覧はフェーズ1ではSalesforce側で対応（新システムのスコープ外）
- 申請者は他者の申請を閲覧不可

### 将来検討（フェーズ2）

EntraIDのユーザーIDをキーにDBへ申請データを紐づけることで、申請者が新システム上で自分の申請ステータスを確認できる画面を追加可能。SF依存を段階的に排除する方向で検討する。

| 項目 | 内容 |
|------|------|
| ユーザー識別 | EntraID の Object ID または UPN をDBのキーとして保存 |
| 申請者画面 | ログインユーザーに紐づく申請のみ表示 |
| ステータス管理 | 情シスが更新 → 申請者がリアルタイムで確認可能 |

## 8. 外部システム連携

### 背景

現行はSalesforce（SF）上のワークフローで機器申請を含む業務全体を管理している。  
本プロジェクトでは機器申請部分を新フォームに切り出すが、SF上に残る別申請・ワークフローとの整合性を保つためデータ連携が必要。

### 連携先一覧

| # | 連携先 | 方向 | タイミング | 用途 |
|---|--------|------|------------|------|
| 1 | Salesforce | 書き込み | 申請完了時 | 申請データをSF側ワークフローへ連携 |
| 2 | マスタデータ（TBD） | 読み取り | フォーム表示時 | 社員情報・納品先マスタの参照 |
| 3 | ITAM | 読み取り／書き込み | フォーム入力時（参照）・申請完了時（書き込み） | 資産情報参照・資産管理入力の自動化 |

### Salesforce

| 項目 | 内容 |
|------|------|
| 連携方向 | 新システム → SF（書き込みのみ） |
| 連携タイミング | 申請完了時のみ |
| 連携API | Salesforce REST API（詳細TBD） |
| 連携データ | 申請内容・申請者情報・機器種別・ステータスなど（詳細TBD） |
| 備考 | SF側に不要な申請が上がらないよう、新システムで申請を制御してから連携する |

### マスタデータ

| 項目 | 内容 |
|------|------|
| 管理場所 | 各エクセルまたは別管理システム（詳細TBD） |
| 連携方向 | マスタ → 新システム（読み取りのみ） |
| 連携タイミング | フォーム表示・入力時 |
| 連携データ | 社員情報・納品先マスタ |

### ITAM

| 項目 | 内容 |
|------|------|
| システム名 | ITAM |
| 連携方向 | 双方向 |
| 読み取りタイミング | フォーム入力時（資産情報を参照してフォームに反映） |
| 書き込みタイミング | 申請完了時（資産管理情報を自動登録） |
| 目的 | 資産管理情報の手入力を排除し入力自動化を実現 |
| 連携API | （詳細TBD） |

### インフラ
