# 申請ツール 非機能要件

---

## 1. 可用性

| 項目 | 内容 |
|------|------|
| 月間稼働率目標 | 99.5%（月間ダウンタイム上限：約3.5時間） |
| 稼働保証時間 | 業務時間（平日 9:00〜16:00） |
| 計画停止ウィンドウ | 平日 21:00〜23:00（最大1時間/回・月1回以内） |
| 緊急停止 | 重大障害・セキュリティインシデント発生時は即時対応（時間帯問わず） |
| 時間外アクセス | 業務時間外の申請受付は可能だが、対応は翌営業日 |

---

## 2. パフォーマンス

機能要件「パフォーマンス要件」セクションに記載。概要は以下のとおり。

| 項目 | 目標値 |
|------|--------|
| 同時接続数 | ピーク時200〜300名 |
| 画面表示速度 | 3秒以内 |
| API応答速度 | 1秒以内 |
| 自動保存API | 500ms以内 |

---

## 3. セキュリティ

機能要件「セキュリティ要件」セクションに記載。概要は以下のとおり。

| 項目 | 内容 |
|------|------|
| 認証 | Microsoft Entra ID（OIDC）・社内VPN限定 |
| 通信 | HTTPS強制・セキュリティヘッダー設定 |
| 入力 | Zodバリデーション・ORM使用（SQLインジェクション防止） |
| アクセス制御 | ロールベース・IDORチェック |

---

## 4. データ管理・保持

| 項目 | 内容 |
|------|------|
| データ保持期間 | 申請データは1年間保持。過去1年分の申請を管理画面から確認可能 |
| アーカイブ | 1年を超えたデータはアクティブDBから別ストレージへアーカイブ |
| 個人情報 | 社員番号・氏名・連絡先等を含むため、アクセスログを記録し権限管理を徹底 |
| データ削除 | 下書き保存データは1週間で自動削除。添付ファイルも合わせて削除 |

---

## 5. バックアップ

| 項目 | 内容 |
|------|------|
| バックアップ頻度 | 毎日1回（深夜帯に自動実行） |
| バックアップ先 | 別ファイルストレージ（詳細TBD） |
| バックアップ形式 | 月ごとにフォルダ分けして保管（例：`backup/2026-04/`） |
| 保持期間 | 1年分（データ保持期間に合わせる） |
| 対象 | DBスナップショット＋添付ファイル（S3等） |
| 失敗時 | バックアップ失敗時は管理者へメール通知 |
| 復元確認 | 四半期に1回、復元テストを実施 |

## 6. メンテナンス計画

### メンテナンスウィンドウ

| 項目 | 内容 |
|------|------|
| 実施時間 | 平日 21:00〜23:00（最大1時間/回） |
| 頻度 | 月1回以内 |

| 作業 | タイミング | 所要時間 |

| ライブラリ・依存関係のバージョンアップ | 月初 | 30〜60分 |

| インフラコスト最適化 | リソース使用状況の見直し |

## 7. 外部システム連携の非機能要件

### タイムアウト設定

| 連携先 | タイムアウト | 備考 |
|--------|------------|------|
| Salesforce | 10秒 | 超過時はリトライキューに積む |
| ITAM | 5秒 | 超過時は手動入力へフォールバック |
| マスタデータ（Excel取り込み） | 30秒 | 超過時は管理者へメール通知 |

### SLA依存・影響範囲

| 連携先 | 影響 | 対応方針 |
|--------|------|---------|
| Salesforce | 申請データの書き込み不可（リトライで吸収） | リトライ最大5回。失敗継続時は管理者通知・手動対応 |
| ITAM | フォーム入力時の資産情報取得不可 | 手動入力へフォールバック。申請自体は続行可能 |
| マスタデータ | 社員情報・納品先の自動補完不可 | 手動入力へフォールバック。申請自体は続行可能 |
| Entra ID | ログイン不可・全ユーザー利用不能 | Entra ID側のSLAに依存。対応不可のため障害情報をユーザーへ周知 |

### API認証管理

| 項目 | 内容 |
|------|------|
| 保管場所 | AWS Systems Manager Parameter Store または Secrets Manager |
| ローテーション | APIキー・トークンの有効期限を管理し、期限1ヶ月前にアラート通知 |
| 通信セキュリティ | 全外部連携はHTTPS必須。証明書検証を必ず行う（スキップ禁止） |

### 連携先の死活監視

| 項目 | 内容 |
|------|------|
| 監視方法 | 定期的（5分間隔）にヘルスチェックAPIへの疎通確認を実施 |
| 異常検知時 | 管理者へメール通知。フォールバック対象の連携先は自動でフォールバックモードへ切り替え |

## 8. 保守性

| 項目 | 内容 |
|------|------|
| 言語 | TypeScript（strict mode）でコード品質を担保 |
| Lint・フォーマット | ESLint + Prettier を CI/CD に組み込み |
| 環境変数 | ローカル開発は `.env.local` で管理。本番環境はAWS Systems Manager Parameter Store または Secrets Manager に登録しコンテナ起動時に自動読み込み。`.env.local` は `.gitignore` に追加してリポジトリに含めない（同じフォルダに置いていてもgitがトラッキングしない）。リポジトリには `.env.local.example`（値なしの雛形）のみ含め、実際の値は別途共有 |
| ドキュメント | READMEにセットアップ手順・環境構成を記載 |
| ログ | 操作ログ・エラーログをCloudWatch等に集約。検索・フィルタが可能な形式で出力 |

## 9. 運用性

| 項目 | 内容 |
|------|------|
| 監視 | AWS CloudWatch等でエラー率・応答速度・CPU使用率を監視 |
| アラート | 閾値超過・障害発生時に管理者へメール通知 |
| デプロイ | CI/CDパイプラインで自動テスト → ステージング → 本番の順にデプロイ |
| ロールバック | デプロイ失敗時は前バージョンへ即時ロールバック |
| 問合せ対応時間 | 業務時間内（9:00〜16:00）。時間外は翌営業日対応 |
