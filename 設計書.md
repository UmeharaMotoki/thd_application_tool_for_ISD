## 1. システム概要

ITサービス依頼（機器貸与・返却）の申請業務を専用Webフォームで受け付け、申請内容をDBに保存しSalesforceへ連携する。申請時のバリデーションで不備を排除し、情報システム部の確認工数を削減することを目的とする。

| 項目 | 内容 |
|------|------|
| 対象業務 | 機器貸与・機器返却（延長含む） |
| 対象ユーザー | 全社員（申請者）・情報システム部（管理者） |
| 認証 | Microsoft Entra ID（OIDC） |
| アクセス制限 | 社内VPN経由のみ |
| 承認フロー | なし（バリデーション通過で自動受付） |

> **PoC実装との差分**: 現在のPoC（仕様書.md）は認証・下書き・ファイル添付・SF/ITAM連携が未実装。本設計書はフル実装を対象とする。

## 3. システム構成・アーキテクチャ

### 3.1 環境構成

| 環境 | 用途 |
|------|------|
| dev（開発） | ローカル開発・単体テスト |
| prod（本番） | 社内VPN経由で全社員が利用 |
| staging（検証） | 必要に応じて追加検討（TBD） |

### 3.2 インフラ構成（AWS）

```
ユーザー（社内VPN）
  ↓ HTTPS
[CloudFront / ALB]
  ↓
[ECS（Next.js コンテナ）]
  ├─ [RDS PostgreSQL]
  ├─ [S3] 添付ファイル・バックアップ・マスタデータ(Excel)
  ├─ [CloudWatch] ログ・監視・アラート
  └─ [Systems Manager Parameter Store / Secrets Manager] 環境変数・APIキー管理

外部連携:
  ├─ Microsoft Entra ID（OIDC認証）
  ├─ Salesforce REST API（申請完了時 書き込み）
  └─ ITAM API（フォーム入力時 読み取り / 申請完了時 書き込み）
```

### 3.3 デプロイ方式

| 区分 | 方式 |
|------|------|
| アプリリリース | ローリングアップデート（ゼロダウンタイム） |
| Next.js本体アップグレード | メンテナンスウィンドウ内（平日21:00〜23:00）で実施 |
| ロールバック | デプロイ失敗時は前バージョンへ即時ロールバック |

## 4. 画面設計

### 4.1 画面一覧

| 画面名 | パス | 対象 | 概要 |
|--------|------|------|------|
| ログイン | `/login` | 全員 | EntraIDへリダイレクト |
| トップ / 申請開始 | `/` | 申請者 | 下書き一覧・新規申請ボタン |
| 事前ヒアリング | `/apply/start` | 申請者 | 申請種別・機器カテゴリ選択 |
| 申請フォーム | `/apply/[type]/[step]` | 申請者 | ステップ形式の申請入力 |
| 申請確認 | `/apply/[type]/confirm` | 申請者 | 入力内容の最終確認 |
| 申請完了 | `/apply/complete` | 申請者 | 完了メッセージ・受付番号表示 |
| 管理ダッシュボード | `/admin` | 管理者 | KPI・申請件数・機器種別集計 |
| 申請一覧（管理者） | `/admin/applications` | 管理者 | 絞り込み・CSV出力 |
| 申請詳細（管理者） | `/admin/applications/[id]` | 管理者 | 申請内容の詳細確認 |
| 機器種別一覧（管理者） | `/admin/devices` | 管理者 | 機器種別ごとの申請一覧 |
| 申請ステータス確認 | `/my/[id]` | 申請者 | フェーズ2で検討 |

### 4.2 画面遷移

```
[未認証] → /login → Entra ID認証 → /（トップ）

[申請フロー]
/ → /apply/start（申請区分・機器カテゴリ選択）
  → /apply/[type]/1（共通項目）
  → /apply/[type]/2（種別固有項目）
  → /apply/[type]/3（条件付き追加項目）
  → /apply/[type]/confirm（確認画面）
  → /apply/complete（完了）

[管理フロー]
/admin → /admin/applications → /admin/applications/[id]
      → /admin/devices
```

### 4.3 ステップウィザード設計

| Step | 内容 |
|------|------|
| Step 1 | 申請区分・機器カテゴリを選択 |
| Step 2 | 共通項目を入力（申請者情報・納品先・希望納品日） |
| Step 3 | 分岐条件項目（社員種別・仕様パターン・機器種別等） |
| Step 4 | 条件付き必須項目を動的表示 |
| Step 5 | 確認画面（全項目表示・各セクションに「修正」リンク） |
| Step 6 | 送信完了（受付番号・次のアクションを明示） |

- ステップ移動はバリデーション通過後のみ「次へ」が有効
- URLは `/apply/[type]/[step]` で同期。ブラウザバック対応
- 未保存変更がある状態でページ遷移しようとした場合は確認ダイアログを表示

### 5.2 共通仕様

| 項目 | 内容 |
|------|------|
| バリデーション | クライアント: React Hook Form + Zod。サーバー: `applicationFormSchema.safeParse` で再検証。サーバー側が正とする |
| 認証確認 | Next.js Middleware でセッション検証。未認証は401を返す |
| ロールチェック | `/api/admin/*` は管理者ロールのみ許可。申請者からは403 |
| レートリミット | 1分間30リクエスト上限（IPベース）。超過時は429 |
| エラーレスポンス | `{ error: string, fieldErrors?: Record<string, string[]> }` |

## 7. 外部システム連携設計

### 7.1 Salesforce連携

| 項目 | 内容 |
|------|------|
| 連携方向 | 新システム → SF（書き込みのみ） |
| タイミング | 申請完了時のみ |
| API | Salesforce REST API |
| 認証情報 | AWS Parameter Store / Secrets Managerに格納 |
| タイムアウト | 10秒 |
| リトライ | 最大5回（指数バックオフ: 1分・5分・15分・30分・60分） |
| 失敗時 | 管理者へメール通知＋手動対応 |
| ユーザー表示 | 「申請は受け付けましたが、連携処理に問題が発生しました。情報システム部へ自動通知済みです。」 |
| 連携データ | 申請内容・申請者情報・機器種別・branch_id（詳細TBD） |

### 7.2 ITAM連携

| 項目 | 内容 |
|------|------|
| 連携方向 | 双方向 |
| 読み取りタイミング | フォーム入力時（資産番号入力後に資産情報を取得・自動補完） |
| 書き込みタイミング | 申請完了時（資産管理情報を自動登録） |
| タイムアウト | 5秒 |
| リトライ | 1回のみ |
| 失敗時 | 手動入力へフォールバック（ITAMから自動入力されるはずだった項目を任意に切り替え）。管理者へメール通知 |
| ユーザー表示 | 「資産情報の取得に失敗しました。手動で入力してください。（入力は任意です）」 |

### 7.3 マスタデータ連携

| 項目 | 内容 |
|------|------|
| 連携方向 | マスタ → 新システム（読み取りのみ） |
| タイミング | バッチ（毎朝4時）または管理者手動実行 |
| ファイル形式 | Excel（.xlsx） |
| 格納場所 | TBD（S3等） |
| タイムアウト | 30秒 |
| 失敗時 | 前回取り込みデータを継続使用。管理者へメール通知 |
| 取り込み対象 | 社員マスタ（employees）・納品先マスタ（delivery_locations） |

### 7.4 Entra ID認証連携

| 項目 | 内容 |
|------|------|
| 方式 | OIDC（OpenID Connect）リダイレクト方式 |
| ライブラリ | NextAuth.js（Auth.js）Azure ADプロバイダー |
| VPN制限 | Entraの条件付きアクセスポリシーでVPN外からのアクセスをブロック |
| 失敗時 | Entra ID側のSLAに依存。障害情報をユーザーへ周知 |

### 7.5 死活監視

| 項目 | 内容 |
|------|------|
| 監視方法 | 5分間隔でヘルスチェックAPIへ疎通確認 |
| 異常検知時 | 管理者へメール通知。フォールバック対象はフォールバックモードへ自動切替 |

## 8. バッチ処理設計

### 8.1 マスタデータ取り込みバッチ

| 項目 | 内容 |
|------|------|
| 実行スケジュール | 毎朝4:00（AWS EventBridge等でトリガー） |
| 実行方式 | ECSタスク（またはLambda: TBD） |
| 処理概要 | S3等の格納場所からExcelファイル取得 → パース → DB上書き更新 |
| 成功時 | 更新件数をmaster_import_logsに記録 |
| 失敗時 | エラーをmaster_import_logsに記録。管理者へメール通知。前回データを継続使用 |
| 手動実行 | 管理画面（`/api/admin/master/import`）から任意タイミングで実行可能 |
| 結果確認 | 管理画面に成功・失敗・更新件数を表示 |

### 8.2 下書き自動削除バッチ

| 項目 | 内容 |
|------|------|
| 実行スケジュール | 毎日深夜（時刻TBD） |
| 処理概要 | `expires_at` が現在時刻より過去のdraftsレコードを削除。対応するattachmentsも削除（S3オブジェクト含む） |

| マスタ取り込み失敗時 | 管理者 | 【申請ツール】マスタ取り込み失敗通知 | 失敗日時・エラー内容 |

| SSL証明書期限1ヶ月前 | 管理者 | 【申請ツール】SSL証明書期限アラート | ドメイン・有効期限 |

## 10. セキュリティ設計

### 10.1 認証・セッション

| 項目 | 内容 |
|------|------|
| 認証 | NextAuth.js（Auth.js）+ Entra ID OIDC |
| セッション有効期間 | 8時間（30分無操作でタイムアウト警告） |
| セッションストレージ | HttpOnly + Secure Cookie（JSからアクセス不可） |
| セッション固定攻撃対策 | ログイン成功時にNextAuth.jsがセッションIDを再発行 |
| ログアウト | Entra ID側のセッションも合わせて破棄 |

### 10.2 通信・ヘッダー

| 項目 | 内容 |
|------|------|
| HTTPS強制 | HTTP → HTTPSリダイレクト |
| セキュリティヘッダー | `Content-Security-Policy` / `X-Frame-Options: DENY` / `X-Content-Type-Options: nosniff` / `Strict-Transport-Security` |
| CSRF対策 | App Router Server Actions は自動処理。API Routesは NextAuth.js セッション検証で対応 |

### 10.3 入力・出力

| 項目 | 内容 |
|------|------|
| XSS対策 | Reactのデフォルトエスケープに依存。`dangerouslySetInnerHTML` 使用禁止 |
| バリデーション | Zodスキーマでサーバー側入力検証。型・長さ・形式を全項目で定義 |
| SQLインジェクション | Prisma ORMのプレースホルダーで対応。生クエリ（`$queryRaw`）は原則禁止。やむを得ず使用する場合は文字列結合によるクエリ構築は絶対禁止 |
| ファイルアップロード | MIMEタイプ・拡張子・ファイルサイズをサーバー側で検証。アップロード先はPresigned URLで一時アクセスのみ |
| エラー情報露出 | 本番環境ではスタックトレース・DB接続情報をレスポンスに含めない |

### 10.4 アクセス制御

| 項目 | 内容 |
|------|------|
| ロールチェック | Next.js Middlewareで全リクエストのロールを検証 |
| IDOR対策 | APIでユーザーIDを必ずセッションから取得。クエリパラメータのIDは信頼しない |
| レートリミット | APIに対してIPベース（1分間30リクエスト上限）。超過時429を返す |
